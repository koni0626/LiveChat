# 記憶機能 仕様

## 目的

各キャラクターが、プレイヤーとの過去の会話・おでかけ・ストーリー体験を覚えているように振る舞えるようにする。

モデル自体を学習させるのではなく、DBに保存した要約記憶を各生成プロンプトへ差し込む。これにより、毎回初対面のような会話ではなく、「このキャラクターとこのプレイヤーの続き」として体験できるようにする。

## 記憶の単位

基本単位は `user_id + character_id` とする。

キャラクターごとに個別記憶を持つ。ノアが覚えていること、ラプラスが覚えていること、導巫女が覚えていることは別管理にする。別キャラクターが、他キャラクターとの会話内容を勝手に知っている状態にはしない。

例:

- ユーザーAに対するノアの記憶
- ユーザーAに対するラプラスの記憶
- ユーザーBに対するノアの記憶

これらはすべて別の記憶として扱う。

将来的に必要であれば、以下の記憶も分けて扱う。

- キャラ別記憶: キャラクターがそのプレイヤーについて覚えていること
- プレイヤープロフィール: 全キャラクターが共有してよいユーザーの呼ばれ名や好み
- ワールド記憶: 世界全体の出来事、ニュース、共通事件

初期実装では、キャラ別記憶とプレイヤーの呼ばれ名を対象にする。

## プレイヤー名

現在、チャットやセッション開始時に名前を入力させているが、これを廃止する。

今後はユーザーごとに「キャラクターから呼ばれる名前」を設定し、チャット・ストーリーセッション・おでかけ・レターなどの会話系生成ではその名前を使う。

### 保存単位

プレイヤー名は `user_id` 単位で保存する。キャラクター別ではなく、ユーザー自身のプロフィールとして扱う。

保存先の案:

- 既存の `user` テーブルに `player_name` を追加する
- または `user_profile` テーブルを追加する

初期実装では、扱う項目が少ないため `user.player_name` 追加でよい。

### 表示・変更UI

ユーザーは、ログアウトボタンがあるユーザーメニュー周辺から自分の名前を変更できる。

想定UI:

- PC: 右上またはサイドバー上部のユーザー情報/ログアウト付近に「プロフィール設定」または「名前を変更」を出す
- スマホ: 下部メニューの「メニュー」内、ログアウト項目の近くに「名前を変更」を出す
- 変更は小さなモーダルで行う
- 入力項目は「キャラクターから呼ばれる名前」
- 保存後、以後の会話生成に即時反映する

### 名前の利用ルール

生成プロンプトには以下のように差し込む。

```text
Player display name: {player_name}
Characters should address the player using this name when natural.
```

ルール:

- キャラクターは、自然な場面で `player_name` を使って呼びかける
- 毎回不自然に名前を連呼しない
- キャラクター固有の二人称設定がある場合は、名前と二人称を自然に併用する
- 名前未設定の場合は、既存の表示名 `display_name` を使う
- それもない場合は「あなた」などの自然な二人称にフォールバックする

### 既存入力の廃止対象

以下の画面・処理で、プレイヤー名の手入力をやめる。

- ライブチャットセッション作成
- チャットルームからのセッション作成
- ストーリーセッション作成
- その他、プレイヤー名を都度入力させている会話開始フォーム

既存DBに保存済みのセッション名やスナップショットはそのまま残してよい。新規作成分からユーザーの `player_name` を使う。

## 保存データ

テーブル案: `character_user_memory`

```text
id
user_id
character_id
relationship_summary
memory_notes
preference_notes
unresolved_threads
important_events
last_interaction_at
created_at
updated_at
```

推奨フィールド:

- `relationship_summary`
  - そのキャラクターから見た、プレイヤーとの関係性要約
- `memory_notes`
  - 一緒に話したこと、共有した出来事、印象的なやりとり
- `preference_notes`
  - プレイヤーの好み、嫌がること、反応が良かった話題
- `unresolved_threads`
  - 次回拾える話題、約束、未回収の流れ
- `important_events`
  - おでかけ完了、ストーリー完了、印象的な選択など
- `last_interaction_at`
  - 最後に記憶更新した日時

JSONカラムにまとめてもよいが、初期実装ではテキスト要約カラム中心が扱いやすい。

## 記憶の作成・更新

生ログを毎回すべて読むのではなく、重要な内容だけを要約して保存する。

更新タイミング:

- ライブチャット: 数ターンごと、またはセッション終了/一定時間後
- おでかけ: 完了時
- ストーリー: セッション進行時、または完了時
- レター: 送信内容を必要に応じて反映

更新フロー:

1. 対象キャラクターとプレイヤーの既存記憶を取得
2. 今回の会話・選択・イベントを短くまとめる
3. AIに「長期記憶へ残す価値があるか」を判定させる
4. 残す場合、既存記憶と今回要約を統合
5. DBへ保存

重要度の低い雑談は保存しない。

## 記憶更新プロンプト

AIには以下のJSONを返させる。

```json
{
  "should_remember": true,
  "relationship_summary": "...",
  "memory_notes": "...",
  "preference_notes": "...",
  "unresolved_threads": "...",
  "important_events": "..."
}
```

ルール:

- 事実と推測を混ぜない
- 一時的な感情を永続的な好みとして保存しない
- センシティブ情報は保存しない、または明示的許可がある場合だけ保存する
- キャラクターが知り得ない情報は保存しない
- 他キャラクターとの会話内容は原則保存しない
- プレイヤー名の変更はキャラ別記憶ではなく、ユーザープロフィール側に保存する

## 利用箇所

各生成プロンプトに `character_user_memory` と `player_name` を差し込む。

対象:

- ライブチャット返答
- ライブチャット画像生成の文脈
- おでかけ本文・選択肢・キャラクター台詞
- ストーリーセッションのGM/キャラクター反応
- レター生成
- 必要ならFeed投稿やワールドニュース

差し込み例:

```text
Player display name:
{player_name}

Character memory about this player:
relationship_summary: ...
shared_memories: ...
player_preferences: ...
open_threads: ...

Use this memory subtly. Do not mention it unnaturally.
```

使い方のルール:

- 毎回露骨に「前に話したよね」と言わせない
- 関係性や話題選びに自然に反映する
- 古い記憶より最近の記憶を少し優先する
- キャラクター性と矛盾する使い方はしない
- プレイヤー名は自然な呼びかけのときだけ使う

## UI

最低限ほしいUI:

- ユーザーメニューからプレイヤー名を変更できる
- キャラクター詳細に「このユーザーとの記憶」表示
- superuser または本人が記憶を確認できる
- 記憶の削除/リセット
- 記憶機能ON/OFF

将来的にあるとよいUI:

- 記憶ごとの編集
- 「この記憶を忘れさせる」
- 記憶の最終更新日時
- 記憶の由来: チャット/おでかけ/ストーリー

## 権限・プライバシー

記憶は `user_id + character_id` に紐づくため、他ユーザーから見えないようにする。

ルール:

- 一般ユーザーは自分に関する記憶のみ閲覧/削除可能
- 管理者は運用上必要な範囲で閲覧可能
- 他ユーザーの記憶をプロンプトに混ぜない
- 他キャラクターの記憶を無断で共有しない
- 記憶機能OFFの場合は保存も参照もしない
- プレイヤー名は同一ユーザー内で共有されるが、他ユーザーには共有しない

## 実装ステップ

1. `user.player_name` 追加、または `user_profile` テーブル追加
2. ユーザーメニュー付近にプレイヤー名変更UIを追加
3. チャット/ストーリー/セッション作成時の名前入力を廃止
4. 会話系プロンプトに `player_name` を差し込む
5. `character_user_memory` モデルとマイグレーション追加
6. `CharacterUserMemoryService` 作成
7. ライブチャットの返答後に記憶更新ジョブを追加
8. ライブチャットプロンプトへ記憶差し込み
9. おでかけプロンプトへ記憶差し込み
10. ストーリープロンプトへ記憶差し込み
11. 記憶表示/削除UI追加
12. 記憶ON/OFF設定追加

最初のMVPは、プレイヤー名のユーザー設定化、ライブチャット記憶、おでかけ記憶差し込みまでで十分とする。
