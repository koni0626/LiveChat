# Feed機能 仕様書

## 1. 目的

LaplaceCity に、X風のキャラクター投稿フィードを追加する。

Feed機能には以下の2つの目的がある。

- キャラクターの宣伝・紹介用コンテンツとして、ユーザーがキャラクターの雰囲気を知れる入口を作る。
- 投稿内容をキャラクター設定の補助情報として扱い、ライブチャット時の性格・話し方・好み・価値観に反映する。

本機能はフォロー、返信、リポストを持たない。  
キャラクターごとの公式ポストを、グローバルなタイムラインとして表示する。

## 2. 用語

| 用語 | 意味 |
|---|---|
| Feed | 全キャラクターの投稿が流れるグローバルなタイムライン |
| Feed投稿 | キャラクター名義で投稿される1件の記事 |
| 投稿キャラクター | Feed投稿の発信者として表示されるキャラクター |
| 投稿管理者 | 投稿を作成・編集・削除できるユーザー。スーパーユーザーまたはワールド管理者 |
| 投稿画像 | Feed投稿に添付される画像。1投稿につき最大1枚 |
| いいね | 一般ユーザーが投稿に対して行う軽いリアクション |
| 人格反映メモ | Feed投稿からAIが抽出した、キャラクター設定へ反映するための要約情報 |

## 3. 体験イメージ

1. 管理者がキャラクターを選ぶ。
2. キャラクターの投稿本文を入力する。
3. 投稿画像をアップロードする、または投稿本文とキャラクター基準画像から生成する。
4. 投稿を公開する。
5. ユーザーはグローバルFeedで投稿を見る。
6. 投稿カードにはキャラクター顔アイコン、キャラクター名、ワールド名、本文、画像、いいね数が表示される。
7. ユーザーは気になるキャラクターやワールドへ移動できる。
8. 投稿本文はキャラクターの人格素材として蓄積され、ライブチャットのプロンプトに要約反映される。

## 4. MVP範囲

- グローバルFeed一覧を表示する。
- キャラクター別に投稿を絞り込める。
- ワールド別に投稿を絞り込める。
- キーワード検索できる。
- 投稿カードに以下を表示する。
  - 顔アイコン
  - キャラクター名
  - ワールド名
  - 投稿本文
  - 投稿画像
  - 投稿日時
  - いいね数
- スーパーユーザー、ワールド管理者が投稿を作成・編集・削除できる。
- 投稿画像をアップロードできる。
- 投稿本文を元に、キャラクター基準画像から投稿画像を生成できる。
- 一般ユーザーがいいねできる。
- 投稿内容から人格反映メモを生成し、ライブチャットで参照できる。

## 5. MVPではやらないこと

- フォロー機能
- 返信
- リポスト
- 引用
- 通知
- ハッシュタグ高度検索
- 投稿予約
- 投稿ごとのコメント欄
- ユーザーがキャラクターに成り代わって自由投稿する機能
- 外部Xへの自動投稿

## 6. 権限

### 6-1. 閲覧権限

Feedはグローバル扱いとする。

- ログインユーザーは公開中のFeed投稿を閲覧できる。
- 未ログイン公開を許可するかは運用方針で決める。MVPではログイン必須を推奨する。
- 非公開ワールドの投稿は、原則Feedに出さない。

### 6-2. 作成・編集・削除権限

投稿を作成・編集・削除できるユーザー:

- スーパーユーザー
- 対象キャラクターが所属するワールドを管理できるプロジェクトユーザー

一般ユーザーは投稿できない。

### 6-3. いいね権限

- ログインユーザーは公開投稿にいいねできる。
- 同一ユーザーが同一投稿に付けられるいいねは1つまで。
- もう一度押すといいね解除する。

## 7. 投稿仕様

### 7-1. 投稿単位

1投稿は以下で構成する。

- キャラクター
- ワールド
- 本文
- 画像0または1枚
- 公開状態
- いいね数
- 作成者
- 作成日時

### 7-2. 投稿本文

- 文字数はMVPでは最大1000字。
- X風の短文を基本にするが、宣伝用として多少長くてもよい。
- MarkdownはMVPでは使わない。改行のみ許可する。
- キャラクターの一人称、口調、価値観が表れる内容を推奨する。

### 7-3. 投稿画像

投稿画像は以下のどちらかで用意する。

- 管理者がアップロードする。
- 投稿本文、キャラクター設定、基準画像からAI生成する。

画像は `asset` として保存し、Feed投稿から参照する。

### 7-4. 公開状態

| 状態 | 意味 |
|---|---|
| `draft` | 下書き。管理者のみ見える |
| `published` | 公開。Feedに表示される |
| `archived` | 非表示。過去投稿として保持 |

MVPのUIでは「下書き」「公開」だけを表示してもよい。  
DB上は将来のため `archived` を残す。

## 8. 画像生成仕様

### 8-1. 目的

投稿本文に合う、キャラクター宣伝用の映える画像を生成する。

### 8-2. 入力

- 投稿本文
- キャラクター名
- キャラクター外見
- キャラクター画風
- キャラクター性格
- キャラクター基準画像
- ワールド名
- 世界観設定
- NGルール

### 8-3. 生成方針

- キャラクターを主役にする。
- 画像内に文字、字幕、吹き出し、署名、ロゴ、透かしを入れない。
- キャラクターの顔、髪、衣装、画風を保つ。
- 投稿本文の雰囲気を絵にする。
- 広告バナーではなく、キャラクターの日常スナップ、公式イラスト、イベントCG風にする。
- プレイヤーは画像に出さない。

### 8-4. 画像サイズ

MVPでは以下を既定とする。

- `1536x1024`

理由:

- Feedカードで横長画像として見やすい。
- ワールド看板画像、ライブチャットの横長表示と相性がよい。

### 8-5. セーフティ

投稿画像生成でも、既存の画像生成セーフティ補正を使う。

- 露骨な性的表現は避ける。
- ただしキャラクター魅力を完全に消さない。
- 画像生成前に、テキストAIで安全かつ魅力が残る画像プロンプトへ言い換える。
- API側で `sexual` などの安全判定に当たった場合は、露骨な語を避けた妥協プロンプトで再生成する。

## 9. キャラクター設定への反映

### 9-1. 基本方針

Feed投稿をそのまま毎回プロンプトへ全投入しない。  
投稿が増えるとプロンプトが重くなり、会話品質も不安定になるため。

代わりに、キャラクターごとに投稿内容を要約した「人格反映メモ」を作る。

### 9-2. 反映対象

Feed投稿から以下を抽出する。

- よく使う言い回し
- 価値観
- 好きなもの
- 嫌いなもの
- 趣味
- こだわり
- 他者への距離感
- 恋愛的な反応傾向
- 世界観内での立ち位置
- 宣伝したいテーマ

### 9-3. 保存先

MVPでは専用テーブル `character_feed_profile` を作る。

理由:

- `character` 本体の設定を勝手に書き換えない。
- 管理者が入力した基本設定と、Feedから抽出した補助情報を分離できる。
- 将来「Feed反映を無効化」しやすい。

### 9-4. 反映タイミング

MVPでは以下のいずれか。

- Feed投稿を保存した直後に、そのキャラクターの人格反映メモを再生成する。
- 管理画面の「Feedから人格メモ更新」ボタンで手動更新する。

推奨MVP:

- 投稿保存後に自動更新する。
- 失敗しても投稿保存は成功扱いにする。
- 失敗時は管理画面に警告表示する。

### 9-5. ライブチャットでの利用

ライブチャットの会話プロンプトに、以下のように追加する。

```text
Feedから見えるキャラクター傾向:
...
```

ただし優先順位は以下とする。

1. 明示的なキャラクター設定
2. キャラクターNGルール
3. ルームのキャラクターへの指示
4. Feed人格反映メモ
5. 直近会話ログ

Feedメモは補助情報であり、明示設定を上書きしない。

## 10. データモデル

### 10-1. `feed_post`

キャラクターのFeed投稿本体。

| カラム | 型 | 必須 | 説明 |
|---|---|---:|---|
| `id` | integer | yes | 主キー |
| `project_id` | integer | yes | 所属ワールド |
| `character_id` | integer | yes | 投稿キャラクター |
| `created_by_user_id` | integer | yes | 投稿を作成した実ユーザー |
| `body` | text | yes | 投稿本文 |
| `image_asset_id` | integer | nullable | 投稿画像 |
| `status` | string | yes | `draft/published/archived` |
| `like_count` | integer | yes | いいね数のキャッシュ |
| `generation_state_json` | text/json | nullable | 画像生成プロンプト、モデル、品質など |
| `created_at` | datetime | yes | 作成日時 |
| `updated_at` | datetime | yes | 更新日時 |
| `published_at` | datetime | nullable | 公開日時 |
| `deleted_at` | datetime | nullable | 論理削除 |

外部キー:

- `project_id -> project.id`
- `character_id -> character.id`
- `created_by_user_id -> user.id`
- `image_asset_id -> asset.id`

インデックス:

- `project_id, created_at`
- `character_id, created_at`
- `status, published_at`

### 10-2. `feed_like`

Feed投稿へのいいね。

| カラム | 型 | 必須 | 説明 |
|---|---|---:|---|
| `id` | integer | yes | 主キー |
| `feed_post_id` | integer | yes | 投稿ID |
| `user_id` | integer | yes | いいねしたユーザー |
| `created_at` | datetime | yes | 作成日時 |
| `deleted_at` | datetime | nullable | 解除日時 |

制約:

- `feed_post_id, user_id` は一意。
- 解除時は `deleted_at` を入れる。
- 再いいね時は `deleted_at` をNULLに戻す。

### 10-3. `character_feed_profile`

Feed投稿から抽出したキャラクター人格メモ。

| カラム | 型 | 必須 | 説明 |
|---|---|---:|---|
| `id` | integer | yes | 主キー |
| `character_id` | integer | yes | 対象キャラクター |
| `profile_text` | text | nullable | ライブチャットへ渡す要約メモ |
| `source_post_count` | integer | yes | 反映した投稿数 |
| `source_latest_post_id` | integer | nullable | 反映した最新投稿ID |
| `summary_state_json` | text/json | nullable | AI要約の補助情報 |
| `created_at` | datetime | yes | 作成日時 |
| `updated_at` | datetime | yes | 更新日時 |

制約:

- `character_id` は一意。

## 11. API設計

### 11-1. Feed閲覧

| Method | Path | 用途 |
|---|---|---|
| `GET` | `/api/v1/feed/posts` | グローバルFeed一覧 |
| `GET` | `/api/v1/feed/posts/{post_id}` | 投稿詳細 |

クエリ例:

| Query | 説明 |
|---|---|
| `character_id` | キャラクターで絞り込み |
| `project_id` | ワールドで絞り込み |
| `q` | 本文検索 |
| `status` | 管理者向け状態絞り込み |
| `limit` | 取得件数 |
| `cursor` | ページング |

一般ユーザーの一覧では `published` のみ返す。  
管理者は自分が管理できるワールドの `draft` も取得できる。

### 11-2. Feed管理

| Method | Path | 用途 |
|---|---|---|
| `POST` | `/api/v1/projects/{project_id}/feed/posts` | 投稿作成 |
| `PATCH` | `/api/v1/feed/posts/{post_id}` | 投稿更新 |
| `DELETE` | `/api/v1/feed/posts/{post_id}` | 投稿削除 |
| `POST` | `/api/v1/feed/posts/{post_id}/image/upload` | 投稿画像アップロード |
| `POST` | `/api/v1/feed/posts/{post_id}/image/generate` | 投稿画像生成 |
| `POST` | `/api/v1/characters/{character_id}/feed-profile/refresh` | Feed人格メモ更新 |

### 11-3. いいね

| Method | Path | 用途 |
|---|---|---|
| `POST` | `/api/v1/feed/posts/{post_id}/like` | いいね |
| `DELETE` | `/api/v1/feed/posts/{post_id}/like` | いいね解除 |

レスポンスには `liked_by_me` と `like_count` を含める。

表示:

### 12-3. 投稿作成・編集画面

## 13. AI仕様

### 13-1. 投稿画像プロンプト生成

投稿本文をそのまま画像生成APIに渡さない。  
一度テキストAIで画像生成用プロンプトに変換する。

入力:

- 投稿本文
- キャラクター設定
- キャラクター基準画像
- ワールド設定

出力JSON:

```json
{
  "image_prompt": "Create a polished character promo illustration...",
  "negative_rules": "No text, no captions, no speech bubbles, no logo, no watermark.",
  "scene_summary": "夜のネオン街で、キャラクターが投稿内容に合う表情を見せている",
  "safety_notes": "露骨な性的表現を避け、魅力は表情と雰囲気で表現する"
}
```

### 13-2. Feed人格メモ生成

キャラクターごとの公開済み投稿から、ライブチャット用の要約を作る。

入力:

- キャラクター基本設定
- 公開済みFeed投稿の本文
- 既存のFeed人格メモ

出力:

```json
{
  "profile_text": "このキャラクターは投稿では、仕事や都市の夜景をよく話題にする。口調は柔らかいが、時々挑発的に距離を詰める...",
  "likes": ["夜景", "丁寧な言葉", "小さな贈り物"],
  "speech_tendencies": ["短い問いかけで相手を誘う", "余韻を残す言い方をする"],
  "conversation_hooks": ["最近の投稿で触れた場所", "お気に入りの小物", "ワールド内イベント"]
}
```

保存時は `profile_text` を主に使う。  
JSON全体は `summary_state_json` に保存してもよい。

## 14. ライブチャットとの連携

### 14-1. 会話プロンプト

キャラクター返信生成時に、対象キャラクターの `character_feed_profile.profile_text` を参照する。

例:

```text
Feedから見えるキャラクター傾向:
このキャラクターは最近、夜景と小さな贈り物に触れる投稿が多い。
ユーザーに対しては、直接説明するより、短い問いかけで興味を引く傾向がある。
```

### 14-2. 画像プロンプト

Feed人格メモは、通常のライブチャット画像生成では優先度を低くする。

画像生成の優先順位:

1. 現在の会話内容
2. 選択中の衣装・基準画像
3. キャラクター外見・画風
4. ワールド設定
5. Feed人格メモ

### 14-3. 投稿からチャットへの導線

Feed投稿カードから以下へ移動できる。

- キャラクターが所属するワールド
- そのキャラクターと話せるルーム一覧

MVPでは、ワールドのルーム一覧へ遷移する。

## 15. エラー処理

- 投稿保存失敗: エラートーストを表示する。
- 画像アップロード失敗: 投稿本文は保持し、画像のみ未設定にする。
- 画像生成失敗: 生成エラーを表示し、再生成できるようにする。
- Feed人格メモ更新失敗: 投稿保存は成功扱いにし、管理者向けに警告を表示する。
- いいね失敗: UIを元の状態に戻す。
