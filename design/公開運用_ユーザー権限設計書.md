# 公開運用 ユーザー権限設計書

## 1. 目的

LaplaceCity を Web 公開するにあたり、利用者ごとのアクセス範囲を明確にする。

本設計では、以下を実現する。

- スーパーユーザーは全データを管理できる
- プロジェクトユーザーは自分が作成したプロジェクトを管理できる
- 一般ユーザーは公開または利用許可されたプロジェクト上で、自分のライブチャットのみ利用できる
- プロジェクトユーザーであっても、一般ユーザーが作成したライブチャット内容を無条件には閲覧できない

## 2. 用語

| 用語 | 意味 |
|---|---|
| スーパーユーザー | システム全体を管理する運営者 |
| プロジェクトユーザー | プロジェクトを作成・編集できる制作者 |
| ユーザー | 既存プロジェクトでライブチャットを作成・利用する利用者 |
| プロジェクト | キャラクター、世界観、画像ルール、ライブチャットの親コンテナ |
| ライブチャット | ユーザーとキャラクターの会話セッション |
| ライブチャット所有者 | そのライブチャットを作成したユーザー |

## 3. ユーザーグループ

### 3-1. スーパーユーザー

システム管理者。

できること:

- 全プロジェクトの閲覧、作成、更新、削除
- 全キャラクター、世界観、固有名詞辞書、画像設定の閲覧、更新、削除
- 全ライブチャットの閲覧、更新、削除
- 全ユーザーの管理
- 利用状況、生成履歴、課金・使用量ログの確認

注意:

- 本番公開時は人数を最小限にする
- 通常利用画面ではなく、管理画面として分離するのが望ましい

### 3-2. プロジェクトユーザー

プロジェクトを作成する制作者。

できること:

- 自分のプロジェクトを作成、閲覧、更新、削除
- 自分のプロジェクト配下のキャラクター、世界観、画像設定、固有名詞辞書を管理
- 自分が作成したプロジェクトのライブチャット一覧を確認
- 自分自身が作成したライブチャットは全文閲覧、更新、削除できる
- 自分のプロジェクト上で作成されたライブチャットの件数、作成日時、使用量などの管理用メタ情報を確認

できないこと:

- 他のプロジェクトユーザーが作成したプロジェクトを見る
- 他のプロジェクトユーザーのキャラクター、世界観、画像設定を編集する
- 自分のプロジェクトであっても、一般ユーザーが作成したライブチャット本文を無条件に閲覧する

補足:

- プロジェクトユーザーはプロジェクトの管理者だが、一般ユーザーの会話内容はプライバシー対象として扱う
- 一般ユーザーのライブチャット本文閲覧が必要な場合は、本人同意、通報対応、管理者権限など別条件を設ける

### 3-3. ユーザー

作成済みプロジェクトでライブチャットを利用する一般利用者。

できること:

- 利用可能なプロジェクトを選択する
- プロジェクト上でライブチャットを作成する
- 自分が作成したライブチャットのみ閲覧、更新、削除する
- 自分のライブチャット内で画像生成、贈り物、基準画像選択などを行う

できないこと:

- プロジェクトを作成する
- キャラクター、世界観、画像設定、固有名詞辞書を編集する
- 他ユーザーのライブチャットを見る
- プロジェクト管理情報を見る

## 4. 権限マトリクス

| 操作 | スーパーユーザー | プロジェクトユーザー | ユーザー |
|---|---:|---:|---:|
| 全プロジェクト閲覧 | 可 | 不可 | 不可 |
| 自分のプロジェクト作成 | 可 | 可 | 不可 |
| 自分のプロジェクト編集 | 可 | 可 | 不可 |
| 利用可能プロジェクト閲覧 | 可 | 可 | 可 |
| キャラクター編集 | 可 | 自分のプロジェクトのみ可 | 不可 |
| 世界観編集 | 可 | 自分のプロジェクトのみ可 | 不可 |
| 固有名詞辞書編集 | 可 | 自分のプロジェクトのみ可 | 不可 |
| ライブチャット作成 | 可 | 自分のプロジェクト上で可 | 利用可能プロジェクト上で可 |
| 自分のライブチャット閲覧 | 可 | 可 | 可 |
| 他ユーザーのライブチャット本文閲覧 | 可 | 原則不可 | 不可 |
| 自分のプロジェクト上のライブチャット一覧 | 可 | メタ情報のみ可 | 自分のもののみ可 |
| 使用量ログ閲覧 | 可 | 自分のプロジェクト集計のみ可 | 自分の利用分のみ可 |
| ユーザー管理 | 可 | 不可 | 不可 |

## 5. データ可視性ルール

### 5-1. プロジェクト

プロジェクトには作成者を持たせる。

基本ルール:

- `superuser`: すべての `project` にアクセス可能
- `project_user`: `project.owner_user_id == current_user.id` のみ管理可能
- `user`: 公開または利用許可された `project` の概要のみ閲覧可能

### 5-2. ライブチャット

ライブチャットには作成者を持たせる。

基本ルール:

- `chat_session.owner_user_id == current_user.id` の場合、本人は全文アクセス可能
- `project.owner_user_id == current_user.id` であっても、他ユーザーの `chat_session` 本文は原則アクセス不可
- `superuser` は全ライブチャットにアクセス可能

プロジェクトユーザーが見られるライブチャット情報:

- セッションID
- プロジェクトID
- 作成日時
- 更新日時
- ステータス
- 話したキャラクター
- メッセージ件数
- 生成画像件数
- 使用量

プロジェクトユーザーが原則見られない情報:

- メッセージ本文
- 贈り物画像
- 生成画像の詳細
- プレイヤー名
- 会話メモ
- 恋愛度評価理由

## 6. DB設計差分

### 6-1. `user`

追加または確認するカラム:

| カラム | 型 | 必須 | 説明 |
|---|---|---:|---|
| `role` | string | 必須 | `superuser`, `project_user`, `user` |
| `status` | string | 必須 | `active`, `suspended`, `deleted` |

既存で類似カラムがある場合は流用する。

### 6-2. `project`

追加または確認するカラム:

| カラム | 型 | 必須 | 説明 |
|---|---|---:|---|
| `owner_user_id` | integer | 必須 | プロジェクト作成者 |
| `visibility` | string | 必須 | `private`, `published`, `unlisted` |
| `chat_enabled` | integer/bool | 必須 | 一般ユーザーがライブチャット作成可能か |

補足:

- `private`: 作成者とスーパーユーザーのみ利用可能
- `published`: 一般ユーザーが一覧から見つけられる
- `unlisted`: URLまたは招待導線を知っているユーザーのみ利用可能

### 6-3. `chat_session`

追加または確認するカラム:

| カラム | 型 | 必須 | 説明 |
|---|---|---:|---|
| `owner_user_id` | integer | 必須 | ライブチャット作成者 |
| `privacy_status` | string | 必須 | `private`, `shared_for_support`, `reported` |

補足:

- 既存の `project_id` と `owner_user_id` の組み合わせでアクセス制御する
- プロジェクトユーザーによる本文閲覧を許可する場合は、`privacy_status` や明示同意テーブルで制御する

### 6-4. 将来候補 `project_access`

プロジェクトごとに利用可能ユーザーを制御する場合に追加する。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | integer | 主キー |
| `project_id` | integer | 対象プロジェクト |
| `user_id` | integer | 利用許可ユーザー |
| `access_type` | string | `viewer`, `chat_user`, `editor` |

MVPでは `project.visibility` と `chat_enabled` のみで開始してよい。

## 7. API認可方針

### 7-1. 共通

すべてのAPIでログイン必須とする。

共通チェック:

1. `current_user` を取得
2. `current_user.status == active` を確認
3. 対象リソースの所有者、プロジェクト所有者、ロールを確認
4. 権限がなければ `403 Forbidden`
5. 存在自体を隠したい場合は `404 Not Found`

### 7-2. プロジェクトAPI

| API | スーパーユーザー | プロジェクトユーザー | ユーザー |
|---|---|---|---|
| `GET /projects` | 全件 | 自分のプロジェクトのみ | 利用可能プロジェクトのみ |
| `POST /projects` | 可 | 可 | 不可 |
| `GET /projects/{id}` | 可 | 自分のプロジェクトのみ | 利用可能プロジェクトの公開情報のみ |
| `PATCH /projects/{id}` | 可 | 自分のプロジェクトのみ | 不可 |
| `DELETE /projects/{id}` | 可 | 自分のプロジェクトのみ | 不可 |

### 7-3. キャラクター・世界観API

プロジェクト編集権限がある場合のみ更新可能。

一般ユーザーはライブチャットに必要な読み取り専用情報のみ取得可能。

### 7-4. ライブチャットAPI

| API | スーパーユーザー | プロジェクトユーザー | ユーザー |
|---|---|---|---|
| `GET /chat/sessions?project_id=` | 全件 | 自分作成分は全文、他者分はメタのみ | 自分作成分のみ |
| `POST /chat/sessions` | 可 | 自分のプロジェクトで可 | 利用可能プロジェクトで可 |
| `GET /chat/sessions/{id}` | 可 | 自分作成分のみ全文可 | 自分作成分のみ |
| `PATCH /chat/sessions/{id}` | 可 | 自分作成分のみ可 | 自分作成分のみ可 |
| `DELETE /chat/sessions/{id}` | 可 | 自分作成分のみ可 | 自分作成分のみ可 |
| `GET /chat/sessions/{id}/messages` | 可 | 自分作成分のみ可 | 自分作成分のみ可 |
| `POST /chat/sessions/{id}/messages` | 可 | 自分作成分のみ可 | 自分作成分のみ可 |
| `POST /chat/sessions/{id}/images/generate` | 可 | 自分作成分のみ可 | 自分作成分のみ可 |

## 8. 画面設計への影響

### 8-1. ログイン・登録

必要な変更:

- 登録時の初期ロールは原則 `user`
- `project_user` はスーパーユーザーが管理画面から作成、または既存ユーザーを昇格して付与する
- `superuser` はDB初期投入または管理コマンドで作成する

### 8-1-1. アカウント作成フロー

公開運用時の推奨フロー:

1. 初期スーパーユーザーを管理コマンドまたは初期データ投入で作成する
2. 一般ユーザーはユーザー登録画面から自分で登録する
3. ユーザー登録画面から作成されるアカウントのロールは必ず `user` に固定する
4. プロジェクトユーザーはスーパーユーザーが管理画面から新規作成する
5. すでに一般ユーザーとして登録済みの人をプロジェクトユーザーにする場合は、スーパーユーザーがロールを `project_user` に昇格する
6. ユーザー自身が登録画面で `project_user` や `superuser` を選択できるUI/APIは提供しない

理由:

- プロジェクトユーザーは画像生成やプロジェクト公開に関わるため、APIコストとコンテンツ管理の責任が大きい
- 自由登録でプロジェクトユーザーを作れるようにすると、不正利用やAPIコスト暴走のリスクが高い
- 一般ユーザーはライブチャット利用に限定することで、公開サービスとして入口を広く保てる

### 8-1-2. ロール付与の操作権限

| 操作 | 実行者 | 備考 |
|---|---|---|
| 初期スーパーユーザー作成 | サーバ管理者 | 管理コマンドまたはDB初期投入 |
| 一般ユーザー登録 | 未ログイン利用者 | 登録画面から作成、ロールは `user` 固定 |
| プロジェクトユーザー新規作成 | スーパーユーザー | 管理画面から作成 |
| 一般ユーザーをプロジェクトユーザーへ昇格 | スーパーユーザー | 管理画面から変更 |
| プロジェクトユーザーを一般ユーザーへ降格 | スーパーユーザー | プロジェクト所有権の扱いを確認してから実行 |
| スーパーユーザー付与 | 原則サーバ管理者 | 画面操作ではなく管理コマンド推奨 |

### 8-2. ダッシュボード

ロール別に表示を変える。

スーパーユーザー:

- 全体管理
- 全プロジェクト
- ユーザー管理
- 使用量確認

プロジェクトユーザー:

- 自分のプロジェクト一覧
- 新規プロジェクト作成
- 自分のプロジェクトの管理情報

ユーザー:

- 利用可能プロジェクト一覧
- 自分のライブチャット一覧

### 8-3. プロジェクト詳細

プロジェクトユーザー:

- キャラクター編集
- 世界観編集
- 画像設定
- ライブチャット利用状況

ユーザー:

- プロジェクト概要
- キャラクター選択
- ライブチャット開始

### 8-4. ライブチャット一覧

プロジェクトユーザー:

- 自分が作成したライブチャット
- 自分のプロジェクトで作成されたライブチャットのメタ情報

ユーザー:

- 自分が作成したライブチャットのみ

## 9. 実装方針

### 9-1. 認可サービスを追加する

`AuthorizationService` を作成し、Blueprint や Service に散らばる権限判定を集約する。

想定メソッド:

```python
can_manage_project(user, project) -> bool
can_view_project(user, project) -> bool
can_create_project(user) -> bool
can_create_chat_session(user, project) -> bool
can_view_chat_session(user, chat_session, *, include_body: bool = True) -> bool
can_manage_chat_session(user, chat_session) -> bool
```

### 9-2. 所有者カラムを必ず保存する

新規作成時:

- `project.owner_user_id = current_user.id`
- `chat_session.owner_user_id = current_user.id`

### 9-3. 一覧APIはロールごとにクエリを分ける

避けるべき実装:

- 全件取得してからPython側で除外する
- フロント側だけで非表示にする

推奨:

- Repository または Service のクエリ時点で絞り込む
- スーパーユーザーのみ全件クエリを許可する

## 10. セキュリティ方針

### 10-1. 会話内容の扱い

ライブチャット本文は個人の会話データとして扱う。

原則:

- 本人とスーパーユーザー以外は閲覧不可
- プロジェクトユーザーでも、他人の会話本文は閲覧不可
- デバッグ目的で見る必要がある場合は、明示同意または通報対応フローを設ける

### 10-2. 生成AIコスト対策

公開時はAPIコスト制御が必須。

必要な制御:

- ユーザーごとの日次生成回数制限
- プロジェクトごとの月次生成回数制限
- 画像生成品質の制限
- スーパーユーザーによる停止措置
- 利用量ログ

### 10-3. URL直打ち対策

すべての詳細画面/APIでサーバ側認可を行う。

フロントでボタンを隠すだけでは不可。

## 11. MVP実装範囲

最初に実装する範囲:

- `user.role` を追加
- `project.owner_user_id` を追加
- `project.visibility` を追加
- `project.chat_enabled` を追加
- `chat_session.owner_user_id` を追加
- プロジェクト一覧のロール別絞り込み
- ライブチャット一覧のロール別絞り込み
- ライブチャット詳細の所有者チェック
- プロジェクトユーザーによる他者ライブチャット本文閲覧禁止

後回しでよい範囲:

- `project_access`
- 招待制
- チーム共同編集
- 本人同意による会話共有
- 管理画面の詳細化
- 課金プラン連動

## 12. 未決事項

実装前に決める必要がある項目:

- 一般ユーザーが利用できるプロジェクトを `published` だけにするか、招待制も同時に入れるか
- プロジェクトユーザーが他者ライブチャットの画像サムネイルを見られるか
- 通報・サポート時に誰が会話本文を見られるか
- 一般ユーザー登録を自由登録にするか、招待制にするか
- 生成AI利用量の上限をロールごとにどう分けるか
