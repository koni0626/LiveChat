# AIノベルゲームツクール アーキテクチャ設計書

## 1. 目的

本書は、AIノベルゲームツクールのアプリケーション全体構造、採用技術、レイヤー分割、モジュール構成、データ保存方針、外部AI連携方針、DBマイグレーション方針を定義する。

本システムは、生成AIを利用してノベルゲームを制作するためのツールであり、作品管理、キャラクター管理、世界観管理、シーン生成、画像生成、プレビュー、エクスポートを行う。

本設計では、**Flask を基盤としたシンプルで拡張しやすい構成** を採用し、MVPを素早く構築できることを重視する。

---

## 2. 設計方針

### 2.1 基本方針

* Flask を使用する
* Blueprint により機能単位でルーティングを分割する
* SQLAlchemy を使用して DB を操作する
* DBスキーマはマイグレーションで管理する
* Controller 層と Service 層を分離する
* 保存ファイルは DB ではなくファイルシステムに保持し、DB ではメタ情報のみ管理する
* AI生成処理は将来的にジョブ化できるような構造にする

### 2.2 目的

* MVP を早く作れること
* 将来的に機能追加しやすいこと
* 画像生成やシーン生成などの責務が混ざらないこと
* ファイル保存ルールが崩れないこと

## 4. 全体構成

## 4.1 論理構成

システムは以下の層に分ける。

1. **Presentation Layer**

   * Flask Blueprint
   * HTTPリクエスト/レスポンス処理
   * 入力受け取りとレスポンス整形

2. **Application / Service Layer**

   * 業務ロジック
   * 作品作成、シーン生成、画像生成、アセット保存などの制御

3. **Domain / Model Layer**

   * SQLAlchemy モデル
   * 業務データ構造

4. **Infrastructure Layer**

   * DBアクセス
   * ファイル保存
   * 外部AI API呼び出し
   * ジョブ管理

### 4.2 レイヤー間ルール

* Blueprint から直接複雑な業務ロジックを書かない
* Controller は Service を呼ぶ
* Service は Model や Infrastructure を利用する
* 外部API呼び出しは専用 Service / Client に閉じ込める
* ファイルパス生成は共通ユーティリティに集約する

## 6. Application Factory パターン

### 6.1 方針

Flask の Application Factory パターンを採用する。

### 6.2 理由

* テストしやすい
* 環境ごとの設定差し替えがしやすい
* 拡張機能を疎結合に初期化できる

### 6.3 役割

`create_app()` で以下を行う。

* Flask app 初期化
* Config 読み込み
* 拡張初期化
* Blueprint 登録
* エラーハンドラ登録
* CLI コマンド登録

## 7. レイヤー設計

## 7.1 Controller Layer

### 役割

* HTTPリクエストを受け取る
* パラメータを取得する
* Service を呼ぶ
* JSONレスポンスを返す
* HTTPステータスを決める

### ルール

* ビジネスロジックは書かない
* DB更新を直接行わない
* 外部AI APIを直接呼ばない

### 例

* `/projects` 作成
* `/scenes/{id}/generate` 実行
* `/assets/upload` 実行

---

## 7.2 Service Layer

### 役割

* 業務ロジックを実装する
* モデル操作をまとめる
* 複数モデル更新の整合性を取る
* 外部API呼び出しを調整する
* 保存パスを決定する

### 主な Service

* `project_service`
* `character_service`
* `scene_service`
* `image_service`
* `asset_service`
* `generation_service`
* `preview_service`

### 例

* 作品作成時に world 初期データを用意する
* scene 生成後に scene_version を追加する
* image 生成後に asset と scene_image を登録する

---

## 7.3 Model Layer

### 役割

* DBテーブルと対応する永続データ構造を定義する
* リレーションを定義する

### 方針

* Flask-SQLAlchemy モデルとして定義する
* 業務ロジックはなるべく持たせすぎない
* 複雑な処理は service に置く

---

## 7.4 Infrastructure Layer

### 役割

* DB接続
* 外部AI API連携
* ファイル保存
* ジョブ管理

### 構成候補

* `clients/` に外部APIクライアント
* `utils/path_manager.py` にパス生成ロジック
* `repositories/` は必要に応じて追加

## 9. DBアクセス方針

## 9.1 ORM

* Flask-SQLAlchemy を使用する
* DBセッションは Flask-SQLAlchemy に委譲する

## 9.2 マイグレーション

* DBスキーマ変更はマイグレーションで管理する
* 手動でDDLを直接本番反映しない
* 開発時も migration ベースでテーブル作成する

### 9.3 採用理由

* スキーマ履歴を追跡できる
* ロールバックしやすい
* チーム開発時の差分管理がしやすい

### 9.4 初期運用

* SQLite3 を利用
* migration ツールで `upgrade` して DB を作る
* 将来的に PostgreSQL へ移行可能なモデル記述を意識する

## 10. 設定管理

## 10.1 config.py

設定は `config.py` に集約する。

### 代表設定

* `SQLALCHEMY_DATABASE_URI`
* `SQLALCHEMY_TRACK_MODIFICATIONS`
* `STORAGE_ROOT`
* `MAX_CONTENT_LENGTH`
* `DEFAULT_IMAGE_QUALITY`
* `DEFAULT_IMAGE_SIZE`
* `TEXT_AI_MODEL`
* `IMAGE_AI_MODEL`
* `OPENAI_API_KEY`

## 10.2 .env による環境変数管理

APIキーや利用モデルなど、環境ごとに変わる値は `.env` ファイルで管理する。

### 主な用途

* OpenAI APIキーの管理
* テキスト生成に使うモデル名の切り替え
* 画像生成に使うモデル名の切り替え
* 開発環境/本番環境での設定切り替え

### 想定する .env 項目

* `OPENAI_API_KEY`
* `TEXT_AI_MODEL`
* `IMAGE_AI_MODEL`
* `SQLALCHEMY_DATABASE_URI`
* `STORAGE_ROOT`
* `DEFAULT_IMAGE_QUALITY`
* `DEFAULT_IMAGE_SIZE`
* `FLASK_ENV`
* `SECRET_KEY`

### .env 記述例

```env
OPENAI_API_KEY=your_openai_api_key
TEXT_AI_MODEL=gpt-5.4-mini
IMAGE_AI_MODEL=gpt-image-1.5
SQLALCHEMY_DATABASE_URI=sqlite:///app.db
STORAGE_ROOT=./storage
DEFAULT_IMAGE_QUALITY=low
DEFAULT_IMAGE_SIZE=1024x1024
FLASK_ENV=development
SECRET_KEY=replace_me
```

## 10.3 config.py での読み込み方針

* `config.py` は環境変数を読み込んで Flask 設定へ反映する
* APIキーはコードに直書きしない
* デフォルト値を持たせつつ、`.env` で上書き可能にする

### 例

* `OPENAI_API_KEY` を外部AIクライアント初期化に利用する
* `TEXT_AI_MODEL` を文章生成系 service で利用する
* `IMAGE_AI_MODEL` を画像生成系 service で利用する

## 10.4 環境切り替え

* DevelopmentConfig
* TestingConfig
* ProductionConfig

### 方針

* 環境変数で差し替え可能にする
* 秘密情報はコードに直書きしない
* `.env` は Git 管理対象に含めない
* `.env.example` を用意して必要なキー一覧だけ共有する

## 11. ファイル保存アーキテクチャ

## 11.1 方針

生成画像・参照画像・エクスポートファイルはファイルシステムに保存し、DBではメタ情報のみ保持する。

## 11.2 保存ルート

`storage/` をルートとし、project 単位に分ける。

### 例

```text
storage/
  projects/
    1/
      reference_images/
        noa_base.png
      generated/
        scenes/
          101/
            scene_full_20260418_163012.png
        characters/
          21/
            smile_20260418_170010.png
      thumbnails/
      exports/
      temp/
```

## 11.3 保存方針

* project ごとに分離する
* scene ごとに生成画像を分離する
* reference_image と generated_image を分ける
* export は専用ディレクトリに保存する
* 一時ファイルは `temp/` に保存し、後で削除可能にする

## 11.4 メリット

* バックアップしやすい
* 削除しやすい
* アセット管理が整理される

---

## 12. パス管理方針

## 12.1 方針

保存先のパス文字列をアプリ各所で直接組み立てない。

## 12.2 実装方針

`utils/path_manager.py` にパス生成を集約する。

### 役割

* project ルート取得
* reference_images ディレクトリ取得
* scene 生成画像保存先取得
* export 保存先取得
* タイムスタンプ付きファイル名生成

### メリット

* 保存先変更に強い
* ルール崩れを防げる
* テストしやすい

## 13. AI連携アーキテクチャ

## 13.1 方針

テキスト生成と画像生成を分離する。

### 理由

* 会話本文をそのまま画像に流し込むと破綻しやすい
* 状態抽出を挟むことで画像生成が安定する
* 各処理を個別に改善しやすい

## 13.2 処理分離

1. シーン本文生成
2. 状態JSON抽出
3. 画像生成プロンプト作成
4. 画像生成
5. 保存と採用管理

## 13.3 クライアント構成

* `text_ai_client.py`
* `image_ai_client.py`

### Service との分担

* `generation_service` が生成の流れを制御
* `image_service` が画像保存と asset 登録を行う
* `scene_service` が scene_version を扱う

---

## 14. シーン生成アーキテクチャ

## 14.1 基本フロー

1. scene 情報を取得
2. world / character / outline 情報を集約
3. scene 生成プロンプトを組み立て
4. テキスト生成AI を呼ぶ
5. scene_version を保存
6. 必要に応じて current scene を更新

## 14.2 状態抽出フロー

1. scene 本文を取得
2. 状態抽出プロンプトを組み立て
3. 状態JSON を生成
4. scene.scene_state_json に保存

---

## 15. 画像生成アーキテクチャ

## 15.1 基本フロー

1. scene_state_json を取得
2. character_image_rule を取得
3. image prompt を作成
4. 画像生成AI を呼ぶ
5. 保存パスを決定する
6. 画像ファイルを保存する
7. asset を登録する
8. scene_image を登録する

## 15.2 ルール

* 画像内にUIテキストを焼き込まない
* low モードを標準にする
* 人物固定要素は毎回明示する
* 参照画像利用時は base_asset を参照する

---

## 16. プレビューアーキテクチャ

## 16.1 目的

編集データをそのままノベルゲーム風に再生できるようにする。

## 16.2 方針

* scene の採用版データを優先する
* scene_image の選択済み画像を表示する
* セリフUIはアプリ側で描画する
* 画像は背景・人物のみを扱う

---

## 17. エラーハンドリング方針

## 17.1 Controller

* 入力エラーをHTTP 400系で返す
* 認証エラーは401/403で返す

## 17.2 Service

* ビジネス例外を独自例外で表現する
* 外部AI APIエラーをラップする
* 保存失敗時のロールバックを行う

## 17.3 共通方針

* スタックトレースをそのままユーザーに返さない
* ログには原因を残す

---

## 18. トランザクション方針

### 対象例

* scene 作成 + scene_choice 作成
* scene_version 採用 + scene.active_version_id 更新
* asset 作成 + scene_image 作成

### 方針

* 複数テーブル更新は service でまとめる
* 失敗時は rollback する
* 外部API呼び出し後の保存処理は特に整合性に注意する

---

## 19. 非同期化方針

## 19.1 MVP

MVPでは同期実行でもよい。

### 対象

* シーン生成
* 状態抽出
* 画像生成

## 19.2 将来拡張

将来的にはジョブキューを導入し、以下を非同期化する。

* シーン生成
* 画像生成
* エクスポート

### 理由

* 画像生成待ち時間をUIから切り離すため
* 再試行や監視をしやすくするため

## 23. まとめ

本システムのアーキテクチャは、**Flask を中心とした軽量で整理しやすいレイヤード構成** を採用する。

重要なポイントは以下。

* Blueprint で controller を分割する
* Service 層に業務ロジックを集約する
* SQLAlchemy + migration で DB を管理する
* ファイル保存先は path_manager に集約する
* シーン生成と画像生成を分離する
* 画像は DB に直接保存せず、asset でメタ管理する

この構成により、MVPを素早く作りつつ、将来的な非同期化やDB移行にも対応しやすい基盤を作る。
