# LiveChat お手紙機能 仕様書

## 1. 目的
- ライブチャットを続けたユーザーに対して、キャラクターからアプリ内DM風の「お手紙」が届く体験を作る。
- 手紙は本物のメール送信ではなく、ユーザーごとのメールボックスに届くアプリ内メッセージとして扱う。
- 会話内容、関係性、評価、贈り物、印象イベントを元に、ユーザーが嬉しくなる手紙本文とイベント画像をAI生成する。
- ユーザーが手紙を読んだ後、自然に「またそのルームへ会いに行く」導線を作る。

## 2. 用語
| 用語 | 意味 |
|---|---|
| お手紙 | キャラクターからユーザー宛てに届くDM風メッセージ |
| メールボックス | ユーザーごとに届いたお手紙を一覧・閲覧する画面 |
| レターイベント | 手紙本文、添付画像、発生元セッション、既読状態をまとめたデータ |
| 添付イベント画像 | 手紙に表示する、キャラクターの映えるイベントCG風画像 |
| 発生元セッション | 手紙生成のきっかけになったライブチャットセッション |
| 戻り先ルーム | 手紙末尾の「また会いに来てね」リンク先になるライブチャットルーム |

## 3. 体験イメージ
1. ユーザーがライブチャットでキャラクターと会話する。
2. 会話後または一定タイミングで、AIが「今なら手紙を書けるか」を判定する。
3. 条件を満たした場合、キャラクターからユーザー宛てにお手紙を生成する。
4. 手紙本文は、直近の会話や印象的な出来事を踏まえて、プレイヤーが嬉しくなる内容にする。
5. 同時に、手紙に添えるイベント画像を生成する。
6. ユーザーのメールボックスに未読のお手紙として表示する。
7. ユーザーが開くと、便箋・封筒・DMのようなモーダルUIで表示する。
8. 手紙の末尾に、発生元ルームまたは該当キャラクターとのライブチャットへ戻るリンクを表示する。

## 4. MVP範囲
- ユーザーごとのメールボックスを作る。
- ライブチャットの会話後に、お手紙生成判定を行う。
- AIが「手紙を書くべき」と判断した場合のみ手紙を生成する。
- 手紙本文をAI生成する。
- 手紙用イベント画像をAI生成し、`asset` として保存する。
- お手紙一覧、未読/既読、詳細モーダルを表示する。
- 手紙からライブチャットルームへ戻れるリンクを表示する。

## 5. MVPではやらないこと
- 実メール送信。
- プッシュ通知。
- ユーザーから手紙へ返信する機能。
- 手紙テンプレートの高度な編集機能。
- 複数キャラクター連名の手紙。
- 課金・消費ポイント連携。

## 6. お手紙生成タイミング
### 6-1. 基本方針
手紙は毎回生成しない。AIが「この会話なら手紙にする価値がある」と判断した場合のみ生成する。

### 6-2. 判定タイミング
MVPでは以下のタイミングで判定する。
- ユーザー発話に対してキャラクターが返信した直後
- 評価スコア更新後
- 贈り物イベント後

### 6-3. 判定条件の例
- 会話が一定回数以上続いている。
- 恋愛度または目的達成度が一定以上上がった。
- 印象的な約束、励まし、贈り物、告白めいたやり取りがあった。
- キャラクターが「後で伝えたいこと」を持てる文脈がある。
- 直近で同じキャラクターから手紙を送っていない。

### 6-4. 抑制条件
- 同一セッションで短時間に連続生成しない。
- 同一ユーザー・同一キャラクターで直近N時間以内に送信済みなら生成しない。
- 会話内容が薄い、事務的、または手紙にする意味が弱い場合は生成しない。
- キャラクター設定やNGルールに反する内容になりそうな場合は生成しない。

## 7. AI判定仕様
### 7-1. 入力
- プロジェクト情報
- ルーム情報
- キャラクター設定
- 発生元セッション情報
- 直近会話ログ
- 評価スコアと評価理由
- 贈り物イベント
- セッション記憶

### 7-2. 出力JSON
```json
{
  "should_send_letter": true,
  "reason": "プレイヤーがキャラクターの弱さを受け止め、関係性が一段進んだため",
  "letter_theme": "今夜の会話を思い出して、少し素直になる手紙",
  "emotional_tone": "甘い、少し照れた、余韻がある",
  "image_direction": "夜の窓辺で、キャラクターが手紙を書き終えて微笑んでいる",
  "cooldown_key": "character:12:user:8:romance-letter"
}
```

### 7-3. `should_send_letter`
- `true`: 手紙本文と画像を生成する
- `false`: 何も作らない

## 8. 手紙本文生成仕様
### 8-1. 入力
- 判定AIの `letter_theme`
- キャラクター設定
- 会話ログ要約
- プレイヤー名
- ルーム名
- キャラクターの一人称、相手の呼び方、話し方、セリフサンプル
- NGルール

### 8-2. 出力JSON
```json
{
  "subject": "さっきのこと、少しだけ書いておきたくて",
  "body": "ぱぱぱへ。\n\nさっき話したこと、まだ少し胸の奥に残っています。\n...",
  "closing_line": "また会いに来てくれたら、続きを話したいです。",
  "cta_label": "また会いに行く",
  "summary": "会話後に照れながら感謝を伝える手紙"
}
```

### 8-3. 文体ルール
- キャラクター本人が書いたようにする。
- 説明文ではなく、手紙文にする。
- キャラクターの一人称、相手の呼び方、口調を守る。
- 会話内容を具体的に1つ以上拾う。
- 「また会いに来てね」系の導線は自然に入れる。
- 過度に長くしない。MVP目安は400から900字。
- システム都合、AI、プロンプト、評価スコアなどのメタ情報は書かない。

## 9. 手紙画像生成仕様
### 9-1. 目的
手紙を開いた時に、ユーザーが「イベントが起きた」と感じられる映えるキャラクター画像を添える。

### 9-2. 画像生成方針
- 発生元キャラクターを主役にする。
- プレイヤーは画像に出さない。原則一人称視点またはキャラクター単独構図。
- 手紙・封筒・スマホ通知・夜の部屋など、手紙らしい演出を使う。
- 文字、字幕、吹き出し、署名、透かしは画像内に入れない。
- キャラクターの基準画像またはセッション基準画像を参照画像として使う。
- 画像サイズと品質はユーザー設定またはシステム既定値に従う。

### 9-3. 画像プロンプト生成
本文生成後、以下を元に画像プロンプトを作る。
- 手紙テーマ
- 手紙本文の感情
- 会話の印象的な出来事
- キャラクター外見
- 現在の世界観
- 手紙を届ける状況

### 9-4. 画像例
- 窓辺で手紙を書き終え、少し照れた顔でこちらを見ている。
- ネオンの街を背景に、封筒を胸元に持って微笑んでいる。
- 贈られたぬいぐるみを横に置いて、便箋に向かっている。
- ベッドサイドの薄明かりで、スマホにDMを送る直前の表情。

## 10. データモデル
### 10-1. `letter`
ユーザー宛てのお手紙本体。

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | integer | yes | 主キー |
| `project_id` | integer | yes | 所属プロジェクト |
| `room_id` | integer | nullable | 戻り先ルーム |
| `session_id` | integer | nullable | 発生元セッション |
| `recipient_user_id` | integer | yes | 宛先ユーザー |
| `sender_character_id` | integer | yes | 送信キャラクター |
| `subject` | string | yes | 件名 |
| `body` | text | yes | 手紙本文 |
| `summary` | text | nullable | 一覧用要約 |
| `image_asset_id` | integer | nullable | 添付イベント画像 |
| `status` | string | yes | `unread/read/archived` |
| `trigger_type` | string | nullable | `conversation/evaluation/gift/manual` |
| `trigger_reason` | text | nullable | 生成理由 |
| `generation_state_json` | text/json | nullable | AI判定・プロンプト・スコア等 |
| `read_at` | datetime | nullable | 既読日時 |
| `created_at` | datetime | yes | 作成日時 |
| `updated_at` | datetime | yes | 更新日時 |
| `deleted_at` | datetime | nullable | 論理削除 |

### 10-2. `letter_generation_guard`
短時間の重複生成を抑制するためのガード。

| カラム | 型 | 必須 | 説明 |
|---|---|---|---|
| `id` | integer | yes | 主キー |
| `recipient_user_id` | integer | yes | 対象ユーザー |
| `sender_character_id` | integer | yes | 対象キャラクター |
| `room_id` | integer | nullable | 対象ルーム |
| `cooldown_key` | string | yes | 判定AIが返す重複抑制キー |
| `last_generated_at` | datetime | yes | 最終生成日時 |

MVPでは `letter` のみでも開始可能。ただし連続生成を避けるため、最低限 `letter` 検索で直近生成を確認する。

## 11. API設計
### 11-1. メールボックス
| Method | Path | 用途 |
|---|---|---|
| `GET` | `/api/v1/letters` | 自分宛てのお手紙一覧 |
| `GET` | `/api/v1/letters/{letter_id}` | お手紙詳細 |
| `POST` | `/api/v1/letters/{letter_id}/read` | 既読化 |
| `DELETE` | `/api/v1/letters/{letter_id}` | 自分のメールボックスから削除またはアーカイブ |

### 11-2. 生成
| Method | Path | 用途 |
|---|---|---|
| `POST` | `/api/v1/chat/sessions/{session_id}/letters/evaluate` | 手紙生成判定を実行 |
| `POST` | `/api/v1/chat/sessions/{session_id}/letters/generate` | 手紙を強制作成。管理・デバッグ用 |

MVPでは、ユーザー発話後の通常返信処理内で内部サービスとして `evaluate` 相当を呼び、画面用APIは後から追加してもよい。

## 12. 画面仕様
### 12-1. メールボックス画面
左メニューまたはトップバーに `お手紙` を追加する。

表示項目:
- 未読件数
- キャラクターサムネイル
- キャラクター名
- 件名
- 一覧用要約
- 受信日時
- 未読/既読表示

操作:
- 手紙を開く
- 既読にする
- アーカイブ/削除
- ルームへ戻る

### 12-2. お手紙詳細モーダル
モーダルUIで表示する。

表示項目:
- 封筒/便箋風ヘッダ
- キャラクター名
- 件名
- 添付イベント画像
- 手紙本文
- 受信日時
- `また会いに行く` ボタン

UIトーン:
- ネオンUIの中に、少し特別感のある便箋・封筒モチーフを混ぜる。
- 画像を上部または右側に大きく表示する。
- 本文は読みやすさ優先で、背景を暗め・文字を明るめにする。

### 12-3. 通知表示
MVPでは以下のいずれか。
- 左メニュー `お手紙` に未読バッジ
- ライブチャット画面上部に「お手紙が届いています」カード
- トーストで「キャラクターからお手紙が届きました」

## 13. 権限
### 13-1. ユーザー
- 自分宛てのお手紙のみ閲覧できる。
- 自分宛てのお手紙を既読化、アーカイブできる。
- 他ユーザー宛てのお手紙は見えない。

### 13-2. プロジェクトユーザー
- 自分宛てのお手紙は通常ユーザーと同じように見える。
- 自分が管理するプロジェクトで発生した他ユーザー宛てのお手紙本文は、MVPでは通常UIから閲覧不可。
- 件数や生成状況などの管理メタ情報のみ将来的に閲覧可能にする。

### 13-3. スーパーユーザー
- 管理・監査目的で全お手紙を閲覧できる設計にできる。
- ただし通常UIでは不用意に本文を見せない。必要なら管理画面を別途作る。

## 14. ライブチャットとの連携
### 14-1. 会話後処理
キャラクター返信後、以下の順で処理する。
1. 会話ログ保存
2. 評価更新
3. セッション記憶更新
4. お手紙生成判定
5. 必要なら手紙本文生成
6. 必要なら手紙画像生成
7. `letter` 保存
8. レスポンスに `new_letter_count` または `new_letter` を含める

### 14-2. 画像生成との関係
- 通常のライブチャット画像とは別に、手紙用画像を生成する。
- `asset.asset_type` は `letter_image` または `generated_image` + metadataで区別する。
- `letter.image_asset_id` に紐づける。
- 手紙画像はライブチャットの現在表示画像には自動反映しない。

### 14-3. ルームへの戻りリンク
手紙には以下のどちらかを保持する。
- `room_id` がある場合: `/projects/{project_id}/live-chat` で該当ルームカードへ誘導
- `session_id` がある場合: `/projects/{project_id}/live-chat/{session_id}` へ直接戻る

MVPでは `session_id` へ戻るリンクを優先する。新しく話し直す導線は次フェーズで追加する。

## 15. AIプロンプト方針
### 15-1. 判定AI
- 手紙を乱発しない。
- ユーザーが嬉しくなる余韻がある時だけ生成する。
- キャラクターの人格と目的に合わない手紙は作らない。
- 会話に具体的な材料がない場合は `should_send_letter=false`。

### 15-2. 本文AI
- キャラクター本人が書いた手紙として作る。
- 会話をそのまま要約しない。
- そのキャラクターが「今だから書ける」内容にする。
- 恋愛目的の場合は、直接的すぎず、キュンとする余韻を優先する。
- 学習目的の場合は、励まし・復習・次に話したいことを自然に入れる。

### 15-3. 画像AI
- 画像内に文字を入れない。
- 手紙本文の内容そのものを絵にしすぎず、感情の余韻を絵にする。
- キャラクターの顔・衣装・雰囲気を保つ。
- ルームや世界観の背景を活かす。

## 16. エラー処理
- 手紙判定AIが失敗した場合: 会話処理は継続し、手紙は生成しない。
- 手紙本文生成が失敗した場合: 手紙は保存しない。
- 画像生成が失敗した場合: 本文だけの手紙として保存するか、MVPでは保存しないかを選ぶ。

推奨MVP:
- 本文生成成功、画像生成失敗の場合は `image_asset_id=null` で保存する。
- UIでは画像なし便箋として表示する。

## 17. 実装フェーズ
### Phase 1: 基盤
- `letter` モデル、Repository、Serviceを追加
- マイグレーション追加
- 自分宛てお手紙一覧API
- 既読API

### Phase 2: UI
- 左メニューに `お手紙`
- メールボックス画面
- お手紙詳細モーダル
- 未読バッジ

### Phase 3: AI生成
- 会話後のお手紙生成判定
- 手紙本文生成
- 手紙画像プロンプト生成
- 手紙画像生成

### Phase 4: 体験調整
- 生成頻度調整
- キャラクター別の手紙傾向
- ルームへの復帰導線
- 手紙イベントの分析・管理

## 18. 未決事項

- プロジェクトユーザーが他ユーザー宛て手紙の本文を管理画面で見られるようにするか。
- メールボックスをグローバル画面にするか、プロジェクト内画面にするか。
- 手紙に返信機能を将来追加するか。
