# ライブ会話モード仕様書（たたき台）

## 1. 目的

既存の `作品 -> 章 -> シーン` 中心の制作フローとは別に、より軽量で直感的な「ライブ会話モード」を提供する。  
利用者はキャラクターの基準画像を与えたうえでチャット形式で会話し、その内容に応じてキャラクターの表情・ポーズ・背景が変化する体験を得る。

本仕様は、既存 `NovelCreator` の資産を流用しつつ、新しい会話中心体験へ段階的に移行するためのたたき台である。

---

## 2. 想定ユーザー体験

### 2-1. やりたいこと

ユーザーは以下の流れで利用する。

1. キャラクターを登録する
2. キャラクターごとに基準画像を設定する
3. 会話セッションを開始する
4. チャット欄から話しかける
5. キャラクターが返答する
6. 会話内容に応じて現在の画像が変化する

### 2-2. 解決したい課題

現行仕様では、以下の負荷が大きい。

- 章、骨子、シーンなどの事前設定が多い
- 画像生成までの導線が長い
- 物語制作ツールとしては強いが、即時に会話を楽しむ用途には重い

ライブ会話モードでは、以下を重視する。

- すぐ始められる
- 入力は会話中心
- 画像変化がすぐ分かる
- 設定画面を最小限にする

## 3. スコープ

### 3-1. MVP に含めるもの

- キャラクターの登録
- キャラクター基準画像の登録
- 会話セッション作成
- 1対1 または少人数の会話
- 会話履歴保存
- 会話から状態を抽出
- 状態に応じた画像生成
- 画像候補の選択
- セッションの途中再開

### 3-2. MVP では後回しにするもの

- 章管理
- 骨子管理
- シーン一覧編集
- 分岐選択肢
- エクスポート
- 長編構成管理
- BGM / 効果音
- セーブスロットの複雑な運用

## 4. 基本コンセプト

### 4-1. 新しい中心概念

旧仕様では `project / chapter / scene` が中心だったが、ライブ会話モードでは以下を中心にする。

- `project`
  - 作品や世界観の入れ物としては残す
- `character`
  - 会話相手
- `chat_session`
  - 1回の会話セッション
- `chat_message`
  - 会話ログ
- `session_state`
  - その時点の背景、表情、ポーズなど
- `session_image`
  - 現在の採用画像

### 4-2. 地の文の扱い

地の文はプレイヤー表示用ではなく、内部処理用の補助データとする。

- 会話生成の補助
- 状態抽出の補助
- 画像生成の補助

プレイヤー向けUIには、原則としてセリフだけを表示する。

## 5. 画面仕様（概要）

### 5-1. ライブ会話画面

最重要のメイン画面。1画面で完結することを重視する。

#### レイアウト案

- 左カラム
  - 会話相手キャラクター一覧
  - セッション一覧
- 中央
  - 現在のメイン画像
  - 必要に応じてテキストボックス表示
- 右カラム
  - 現在状態
  - 背景
  - 表情
  - ポーズ
  - 時間帯
  - 直近の画像候補
- 下部
  - チャット履歴
  - 入力欄
  - 送信ボタン

#### 主要操作

- 発話送信
- 画像更新
- 画像候補切り替え
- テキスト表示/非表示
- セッション名変更

### 5-2. キャラクター管理画面

現行のキャラクター編集画面を簡略化して利用する。

必要項目:

- 名前
- 役割
- 話し方
- 性格
- 外見要約
- 基準画像

### 5-3. セッション一覧画面

会話セッションの新規開始・再開に使う。

必要項目:

- セッション名
- 相手キャラクター
- 最終更新日時
- 現在画像のサムネイル

## 6. 会話フロー

### 6-1. 基本フロー

1. ユーザーが発話を送信
2. 発話を `chat_message` として保存
3. AI が返答文を生成
4. 返答文を `chat_message` として保存
5. 会話全体から `session_state` を更新
6. 状態変化が大きい場合のみ画像生成
7. 新画像を表示

### 6-2. 画像更新条件（MVP案）

毎発話で画像生成すると重いため、以下の条件で更新する。

- 背景が変わった
- 表情が大きく変わった
- ポーズが変わった
- シーン転換に相当する会話になった
- 手動で再生成が押された

### 6-3. 画像生成の材料

画像は全文脈ではなく「今見せたい1カット」を材料にする。

利用する入力:

- 直近の会話
- 内部用地の文 / 描写メモ
- `session_state`
- 描画対象キャラクター
- 基準画像

利用しない方向:

- 全会話履歴全文
- 無関係な登場キャラ全員
- 長い骨子本文の丸投げ

## 13. MVPの推奨初期仕様

初手は以下を推奨する。

- 1 project に対して複数 chat_session
- 1 session は最大2キャラ会話
- プレイヤー入力あり
- 画像更新は手動 + 重要変化時のみ
- 画像は最大2キャラ構図
- プレビュー表示はセリフのみ

これにより、実装量を抑えつつ、求める体験に最短で近づける。

## 14. 会話アルゴリズム再設計

ライブ会話では、最終的なセリフだけを直接作らず、会話の意図と進行を先に決めてからセリフを書く構成を目指す。

### 14-1. 理想のレイヤ構成

1. `Player Input`
   - プレイヤー発話を受け取る
2. `Conversation Director`
   - このターンの目的を決める
   - 例: `invite` `tease` `reveal` `test` `comfort` `escalate`
3. `Scene Progression`
   - 場面が進んだか、場所や背景が変わったかを決める
4. `Relationship / Emotion Update`
   - 好感・興味・信頼・警戒・緊張などの変化を決める
5. `Line Writer`
   - 上記の結果とキャラクター設定を見て、最終的なセリフを書く
6. `Visual Direction`
   - 画像にするなら、今の見せ場を短い画像向け指示へ変換する

### 14-2. 現在の実装状況（v1）

現在は v1 として、次の要素まで入っている。

- セッション開始時にキャラクター側から最初の一言を出す
- `scene_progression` を生成し、`scene_phase` / `location` / `background` / `focus_summary` / `next_topic` / `transition_occurred` を `session_state.state_json` に保存する
- 返信生成時に `scene_progression` を参照し、同じ案内や同じ提案の繰り返しを減らす
- 画像生成前に `visual_direction` を生成し、会話から画像用の見せ場を切り出す
- `first_person` / `second_person` / `speech_style` / `speech_sample` を会話生成へ渡し、人称や口調の崩れを抑える

### 14-3. 今後の再設計方針（v2）

v2 では、返信AIを単体で改善するのではなく、会話そのものを役割分担で組み直す。

- `Conversation Director` を追加して、毎ターンの `turn_intent` を先に決める
- `Relationship / Emotion Update` を追加して、会話で何が心情的に動いたかを保存する
- `Line Writer` を独立させ、Director と状態更新の結果だけを材料にセリフを書く
- キャラクターごとに「何を望んでいるか」「どう相手を引っ張るか」を別データで持つ
- 毎ターン `新情報 / 感情変化 / 関係変化 / 場面進行` のいずれかを必ず起こす
- 画像生成も Director の判断を参照し、会話・場面・画像を同じ意図で進める

この再設計により、今の「無難で受け身な返答」から、
- キャラクターが自分の意図で会話を引っ張る
- 場面がきちんと進む
- 関係性の変化が会話に出る
- 画像もその進行に追従する

という形へ移行する。
